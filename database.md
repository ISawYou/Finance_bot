# Database

## Общая логика

База данных строится вокруг трех ключевых сущностей:

1. **Obligation** — обязательство компании  
2. **Scheduled Payment** — плановый платеж по обязательству  
3. **Bank Transaction** — фактическая банковская операция  

На этой связке строится логика продукта:

- обязательство отвечает на вопрос: **что компания обязана платить**
- плановый платеж отвечает на вопрос: **когда и сколько нужно оплатить**
- банковская операция отвечает на вопрос: **что реально произошло по счету**

Дополнительно вводится сущность **Payment Match**, которая связывает план и факт.

---

## Цели БД

База должна позволять:

- хранить реестр всех финансовых обязательств компании;
- автоматически раскладывать обязательные платежи вперед по датам;
- хранить фактические операции из банковских выписок;
- сопоставлять плановые платежи с фактическими списаниями;
- поддерживать платежный календарь;
- давать основу для контроля денег для собственника и автоматизации операционки для финдира.

---

## Основные принципы проектирования

### 1. Не смешивать план и факт
Обязательство, плановый платеж и банковская операция — это разные сущности.  
Их нельзя хранить в одной таблице.

### 2. Supabase — единый источник правды
После загрузки и обработки все данные должны храниться в Supabase.  
Google Sheets и банковские выписки — только внешние источники.

### 3. Поддержка роста
Схема должна позволять в будущем добавить:
- несколько юрлиц;
- несколько банковских счетов;
- web-панель;
- роли и права;
- прогноз cash flow;
- управленческую аналитику.

### 4. Прозрачность изменений
Критичные изменения должны логироваться через audit log.

- 05.07.2026

**что именно нужно оплатить и когда?**

# Связи между таблицами

## Основные связи

### obligations → scheduled_payments
Одно обязательство может иметь много плановых платежей.

Связь:
- `scheduled_payments.obligation_id` → `obligations.id`

### bank_statement_uploads → bank_transactions
Одна выписка может содержать много банковских операций.

Связь:
- `bank_transactions.statement_upload_id` → `bank_statement_uploads.id`

### scheduled_payments ↔ bank_transactions
Связь many-to-many через `payment_matches`.

Это нужно потому, что:
- один плановый платеж может быть закрыт одной операцией;
- в будущем возможны частичные оплаты;
- одна операция теоретически может покрывать несколько обязательств.

### users
Используется в таблицах:
- `obligations.created_by`
- `bank_statement_uploads.uploaded_by`
- `payment_matches.confirmed_by`
- `audit_log.actor_user_id`

### counterparties и categories
Используются как справочники в `obligations`.
